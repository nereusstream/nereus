# 01 API and Domain Model

本文定义 Phase 1 需要演进的 Java API、value object、错误语义和包边界。目标是让后续实现
可以直接按这里拆类，而不是在写代码时重新发明 API。

## 1. API Principles

1. API 是 protocol-neutral L0 storage API，不包含 Pulsar 或 Kafka 类型。
2. 所有异步方法返回 `CompletableFuture`，实现不应在调用线程同步抛业务异常。
3. `append` 的成功结果代表 WAL durable、Oxia stream-head commit 已完成，且本 slice 的 offset index
   已物化到可读路径。
4. `read` 正常读取 Oxia offset index 已发布的 range；如果 stream head 已提交但 index 物化缺失，
   实现必须先从 commit-log 修复或报告 metadata invariant。
5. offset 使用 half-open range `[startOffset, endOffset)`。
6. Phase 1 允许 payload opaque，但必须保存 record count、entry count、payload format、entry index ref。

## 2. Target `StreamStorage`

Package:

```text
io.nereus.api
```

Target interface:

```java
public interface StreamStorage extends AutoCloseable {
    CompletableFuture<StreamMetadata> createOrGetStream(
            StreamName streamName,
            StreamCreateOptions options);

    CompletableFuture<AppendSession> acquireAppendSession(
            StreamId streamId,
            AppendSessionOptions options);

    CompletableFuture<AppendResult> append(
            StreamId streamId,
            AppendBatch batch,
            AppendOptions options);

    CompletableFuture<ReadResult> read(
            StreamId streamId,
            long startOffset,
            ReadOptions options);

    CompletableFuture<ResolveResult> resolve(
            StreamId streamId,
            long startOffset,
            ResolveOptions options);

    CompletableFuture<Void> trim(
            StreamId streamId,
            long beforeOffset,
            TrimOptions options);

    CompletableFuture<StreamMetadata> getStreamMetadata(StreamId streamId);

    @Override
    void close();
}
```

Implementation status:

- M0 removed the Phase 0 `StreamStorage.append(StreamId, byte[], int)` and
  `read(StreamId, long, int)` skeleton methods.
- The full Phase 1 API shape now exists in `nereus-api`.
- A byte-array convenience wrapper can be reintroduced later only as a default wrapper over this full API,
  not as a separate durable surface.

Public offset argument validation:

- `read(streamId, startOffset, options)` and `resolve(streamId, startOffset, options)` require
  `startOffset >= 0`；
- `trim(streamId, beforeOffset, options)` requires `beforeOffset >= 0`；
- negative public offsets complete the returned future with `INVALID_ARGUMENT`, except for null arguments
  or other programmer mistakes that may fail before future creation。

### `StreamCreateOptions`

```java
public record StreamCreateOptions(
        StorageProfile profile,
        Map<String, String> attributes) {
}
```

Phase 1 accepts only `StorageProfile.OBJECT_WAL`.
`attributes` are opaque stream-level metadata stored in `StreamMetadataRecord` and returned by
`getStreamMetadata`. They are not used for partitioning, visibility, or object keys.

Validation:

- `profile == StorageProfile.OBJECT_WAL` in Phase 1；
- attribute keys and values must be non-null UTF-8 strings；
- Phase 1 enforces `MAX_STREAM_ATTRIBUTES_ENCODED_BYTES = 16 KiB` for the total encoded map size because
  attributes live in metadata values；
- records must defensively copy the attributes map.

### `AppendSessionOptions`

```java
public record AppendSessionOptions(
        String writerId,
        Duration ttl,
        boolean allowStealExpiredSession) {
}
```

`allowStealExpiredSession` does not allow stealing a live session. It only makes the caller's intent
explicit when the metadata record is already expired.

Validation:

- `writerId` must be non-blank；
- `ttl` must be positive；
- implementation may enforce configured min/max TTL bounds and fail with `INVALID_ARGUMENT` when the
  requested value is outside those bounds.

## 3. Identity Types

### `StreamId`

M0 keeps the existing record and adds Phase 1 generation rules through shared helpers:

```java
public record StreamId(String value) {
}
```

Rules:

- non-null, non-blank；
- contains only a key-safe value generated by the metadata layer in normal production use；
- stable once created；
- generated by metadata layer when `createOrGetStream` creates a new stream；
- Phase 1 uses deterministic generation from the exact `StreamName.value()` so stream creation can use one
  authoritative stream-head conditional put without random id allocation；
- not derived from object key；
- not tied to Pulsar topic name.

### `StreamName`

`StreamName` is an opaque logical name used for idempotent create-or-get.

```java
public record StreamName(String value) {
}
```

Examples:

```text
tenant/ns/topic-partition-0
test-stream-a
```

Phase 1 treats the value as opaque and exact. It must not parse tenant/namespace/topic syntax, normalize
case, collapse separators, or keep a second external topic identity in L0. Future 2 can map Pulsar topic
identity into the `StreamName` value before calling this API.

Canonicalization rule:

- trim leading/trailing whitespace at API boundary only if the caller explicitly normalizes before
  construction; `StreamName` itself should treat the provided string as exact；
- reject blank values；
- use UTF-8 bytes of the exact value for `streamNameHash` and deterministic stream id generation；
- the durable `StreamId` is `s-` plus full `base32lower_nopad(sha256(UTF-8 exact value))`; readable
  aliases belong in stream attributes, not in the durable id。

### Shared Key and Hash Helpers

Path and identity helpers used by more than one Phase 1 module must live below `nereus-api` so
`nereus-metadata-oxia` and `nereus-object-store` cannot drift:

```text
io.nereus.api.keys
  KeyComponentCodec
  DeterministicIds
```

Required helper contracts:

```java
public final class KeyComponentCodec {
    public static String encodeComponent(String value);
    public static String encodeNonNegativeLong(long value);
}

public final class DeterministicIds {
    public static StreamId streamIdFor(StreamName name);
    public static String streamNameHash(StreamName name);
    public static String stableHashComponent(String value);
    public static String randomRunIdHash(byte[] randomBytes);
}
```

Rules:

- `KeyComponentCodec.encodeComponent` is the durable `keyComponent(x)` helper used by Oxia keyspace and
  object key generation；
- a raw component is allowed only when it contains ASCII letters, digits, `_`, `-`, or `.`, is non-empty,
  is not `.` or `..`, does not start with the reserved prefix `b32-`, and does not match a Windows drive
  designator such as `C:`. Every other value must be encoded；
- `encodeNonNegativeLong` returns the 19-digit zero-padded decimal encoding used by offset and generation
  keys and rejects negative values；
- `streamIdFor` and `streamNameHash` use the exact UTF-8 bytes of `StreamName.value()`；
- `stableHashComponent` and `randomRunIdHash` return key-safe `base32lower_nopad(sha256(...))`
  components unless a later document defines a different collision budget and migration plan；
- `randomRunIdHash` input must be generated from a cryptographically strong random source with at least
  128 bits of entropy per `DefaultStreamStorage` process incarnation；
- modules must not duplicate these algorithms in local utility classes. Tests should compare metadata and
  object-store outputs from the same shared helper.

### Shared Metadata Canonicalization Helpers

M1 adds these protocol-neutral helpers in `nereus-api`:

```text
io.nereus.api.ApiLimits
io.nereus.api.MetadataCanonicalizer
```

`ApiLimits` currently defines:

```java
MAX_STREAM_ATTRIBUTES_ENCODED_BYTES = 16 * 1024
MAX_ENTRY_ATTRIBUTES_ENCODED_BYTES = 16 * 1024
MAX_SCHEMA_REFS_ENCODED_BYTES = 16 * 1024
```

`MetadataCanonicalizer` is the shared API-boundary helper for metadata values that will later be copied
into WAL and Oxia records:

- maps are defensively copied into unmodifiable insertion-ordered maps sorted by UTF-8 key bytes；
- map encoded-size guard uses `4-byte key length + key UTF-8 bytes + 4-byte value length + value UTF-8
  bytes` per entry；
- `schemaRefs` are defensively copied, sorted by `(namespace UTF-8 bytes, id UTF-8 bytes, version)`,
  reject duplicate tuples, and use `4-byte namespace length + namespace UTF-8 bytes + 4-byte id length +
  id UTF-8 bytes + 8-byte version` per item for the API encoded-size guard；
- this size guard is an API invariant, not a promise that the final metadata codec has no envelope or
  field-tag overhead. Future codec work may choose Protobuf or another schema-first encoding, but it must
  preserve the same canonical order and 16 KiB public limits unless this document is changed first.

### `OffsetRange`

Existing record is correct as the shared half-open range type:

```java
public record OffsetRange(long startOffset, long endOffset) {
    public long recordCount() {
        return endOffset - startOffset;
    }
}
```

Additional Phase 1 helper methods:

```java
boolean isEmpty();
boolean contains(long offset);
boolean overlaps(OffsetRange other);
```

These helpers should stay pure and allocation-free.

Validation:

- `startOffset >= 0`；
- `endOffset >= startOffset`；
- offset arithmetic in append and trim code must use overflow-checked operations such as
  `Math.addExact` before constructing an `OffsetRange`。

| --- | --- | --- | --- | --- | --- |

into `SEALED`, `DELETING`, or `DELETED`.

## 5. Append Domain

### `AppendBatch`

```java
public record AppendBatch(
        PayloadFormat payloadFormat,
        List<AppendEntry> entries,
        int recordCount,
        int entryCount,
        long minEventTimeMillis,
        long maxEventTimeMillis,
        List<SchemaRef> schemaRefs,
        Map<String, String> projectionHints,
        Optional<Checksum> checksum) {
}
```

Phase 1 intentionally omits producer identity and producer sequence from the public L0 append API. A caller
retry after losing the append result can duplicate data unless it reuses the same in-memory physical slice
attempt. Future protocol layers will add producer-level dedup using Pulsar/Kafka-specific identity.

Validation:

- `entries` must not be empty；
- `entries` must not contain nulls；
- `recordCount > 0`；
- `entryCount == entries.size()`；
- sum of `AppendEntry.recordCount()` must equal `recordCount`；
- `minEventTimeMillis >= 0` and `maxEventTimeMillis >= minEventTimeMillis`；
- every entry `eventTimeMillis` must be within `[minEventTimeMillis, maxEventTimeMillis]`；
- `AppendBatch.checksum` is optional caller payload integrity. If present in Phase 1, it must be CRC32C
  over the concatenation of `AppendEntry.payload` bytes in append order. Mismatch fails before WAL upload；
- `schemaRefs` must not contain nulls or duplicate `(namespace,id,version)` tuples. Phase 1 canonicalizes
  them by `(namespace UTF-8 bytes, id UTF-8 bytes, version)` and enforces
  `MAX_SCHEMA_REFS_ENCODED_BYTES = 16 KiB` because they are copied into WAL metadata and offset index
  values；
- Phase 1 public append requires `projectionHints` to be empty. Future protocol layers may define a
  durable mapping from hints to `ProjectionRef`, but Phase 1 must not silently drop or invent that mapping；
- Phase 1 public `append` accepts only `PayloadFormat.OPAQUE_RECORD_BATCH`；
- `OPAQUE_RECORD_BATCH` requires every `AppendEntry.recordCount() == 1` because Phase 1 has no
  protocol decoder for sub-record byte boundaries。

### `AppendEntry`

```java
public record AppendEntry(
        byte[] payload,
        int recordCount,
        long eventTimeMillis,
        Map<String, String> attributes) {
}
```

Phase 1 stores payload bytes opaquely. It does not decode Pulsar batch metadata or Kafka record batch
headers.

Validation:

- `payload` must be non-null. Empty payload is allowed and still consumes one stream offset for
  `OPAQUE_RECORD_BATCH`；
- `eventTimeMillis >= 0`；
- `attributes` must not contain null keys or values and Phase 1 enforces
  `MAX_ENTRY_ATTRIBUTES_ENCODED_BYTES = 16 KiB` per entry；
- records must defensively copy `payload` and `attributes` at the API boundary.

Record count rule:

- `recordCount == 1` for `OPAQUE_RECORD_BATCH`；
- `recordCount > 1` is allowed only for a payload format whose reader can reconstruct record boundaries
  or whose protocol projection can legally return an entry-level batch with batch indexes；
- because Phase 1 has no such decoder/projection, any `recordCount > 1` entry and any non-opaque public
  append must fail before WAL upload。

### `PayloadFormat`

```java
public enum PayloadFormat {
    OPAQUE_RECORD_BATCH,
    PULSAR_ENTRY_BATCH,
    KAFKA_RECORD_BATCH,
    NORMALIZED_ROW_BATCH
}
```

Phase 1 public append accepts only `OPAQUE_RECORD_BATCH`. The other enum values are reserved protocol
labels for later futures; they must not cause Phase 1 modules to depend on Pulsar, Kafka, or row-format
classes. If a Phase 1 reader/resolver encounters a committed offset-index record with a non-supported
payload format, it should fail with `UNSUPPORTED_FORMAT` instead of trying to decode or silently treating
it as opaque.

### `AppendOptions`

```java
public record AppendOptions(
        Optional<AppendSession> appendSession,
        DurabilityLevel durabilityLevel,
        Duration timeout,
        boolean autoAcquireSession,
        Map<String, String> tags) {
}
```

`tags` are local diagnostics/metrics hints only. Phase 1 must not persist them in WAL objects or Oxia
metadata, and they must not affect partitioning, visibility, object keys, or retry semantics. Tags must
have non-null UTF-8 keys/values and should stay small enough for in-memory logging.

`durabilityLevel` for Phase 1:

```java
public enum DurabilityLevel {
    WAL_DURABLE_AND_INDEX_COMMITTED
}
```

### `AppendSession`

```java
public record AppendSession(
        StreamId streamId,
        String writerId,
        long epoch,
        String fencingToken,
        long leaseVersion,
        long expiresAtMillis) {
}
```

Rules:

- generated and renewed through Oxia；
- `writerId` must be unique per running writer process；two live processes sharing one `writerId`
  intentionally bypass session ownership transfer and can only be protected by offset CAS conflicts；
- `fencingToken` is opaque；
- caller may pass it back in `AppendOptions`；
- stale token must fail before stream-head CAS commits a new visible range。

### `AppendResult`

Existing `AppendResult` should expand:

```java
public record AppendResult(
        StreamId streamId,
        OffsetRange range,
        long committedEndOffset,
        long generation,
        ObjectId objectId,
        ObjectKey objectKey,
        String sliceId,
        long objectOffset,
        long objectLength,
        PayloadFormat payloadFormat,
        int recordCount,
        int entryCount,
        long logicalBytes,
        List<SchemaRef> schemaRefs,
        EntryIndexRef entryIndexRef,
        Checksum objectChecksum,
        Checksum sliceChecksum,
        Optional<ProjectionRef> projectionRef,
        long commitVersion) {
}
```

Rules:

- `range.startOffset()` and `range.endOffset()` come from Oxia commit result；
- `committedEndOffset == range.endOffset()` for simple append；
- `generation == 0` for Phase 1 WAL appends；
- `objectOffset/objectLength` must be non-negative and `objectOffset + objectLength` must not overflow；
- `recordCount > 0`, `entryCount > 0`, and `recordCount == range.recordCount()`；
- `sliceId`, checksums, and object location come from the durable WAL write result and are useful for
  diagnostics, repair, and idempotent same-slice replay；
- `logicalBytes` is the caller-visible uncompressed payload bytes represented by this append. For
  `OPAQUE_RECORD_BATCH` Phase 1 tests it is the sum of `AppendEntry.payload().length`；
- `schemaRefs` are canonicalized from `AppendBatch.schemaRefs` and copied through WAL metadata and offset
  index. They are slice-level metadata only and are not ordering truth；
- `objectChecksum` is the WAL canonical object checksum stored in the object header. It is not the object
  store checksum over exact uploaded bytes；
- Future compaction must not change prior append result offsets；
- Phase 1 public append returns `projectionRef=Optional.empty()`；`projectionRef` is not ordering truth。

## 6. Read and Resolve Domain

### `ReadOptions`

```java
public record ReadOptions(
        int maxRecords,
        int maxBytes,
        ReadIsolation isolation,
        Duration timeout) {
}
```

Validation:

- `maxRecords > 0`；
- `maxBytes > 0`；
- `maxBytes` counts payload bytes returned to the caller, not entry-index bytes or metadata reads。

Limit semantics:

- a read may return zero-byte entries even after the positive-byte budget has reached exactly zero, as long
  as `maxRecords` is not exhausted；
- once the byte budget is zero, the next positive-length entry stops the read without returning that entry；
- if no entry has been returned and the first readable positive-length entry is larger than `maxBytes`, the
  read fails with `READ_LIMIT_TOO_SMALL`。

Phase 1 isolation:

```java
public enum ReadIsolation {
    COMMITTED
}
```

### `ReadResult`

```java
public record ReadResult(
        StreamId streamId,
        long requestedOffset,
        long nextOffset,
        List<ReadBatch> batches,
        boolean endOfStream) {
}
```

`endOfStream` means no committed entry currently covers `requestedOffset`. It is not a permanent EOF.

### `ReadBatch`

```java
public record ReadBatch(
        OffsetRange range,
        PayloadFormat payloadFormat,
        byte[] payload,
        List<SchemaRef> schemaRefs,
        EntryIndexRef entryIndexRef,
        Optional<ProjectionRef> projectionRef,
        ObjectId sourceObjectId,
        long sourceObjectOffset,
        long sourceObjectLength) {
}
```

Rules:

- `range` is the logical offsets represented by `payload` after clipping to the request；
- `schemaRefs` are copied from the committed slice metadata and are not decoded from payload bytes；
- `sourceObjectOffset` and `sourceObjectLength` describe the exact physical bytes returned in `payload`,
  not necessarily the full committed stream slice；
- `sourceObjectOffset/sourceObjectLength` must be non-negative and `sourceObjectOffset +
  sourceObjectLength` must not overflow；
- `sourceObjectLength == payload.length` for uncompressed `OPAQUE_RECORD_BATCH` Phase 1 tests；
- `entryIndexRef` is the source committed slice's entry-index reference. It is not a newly encoded clipped
  entry index for this `ReadBatch`；
- Phase 1 `OPAQUE_RECORD_BATCH` read results must return one `ReadBatch` per opaque append entry. The
  reader must not concatenate multiple opaque entry payloads into one `ReadBatch` because there is no
  protocol decoder or delimiter that lets callers recover entry boundaries；
- if a read would need to return non-contiguous physical entry bytes, it must return multiple
  `ReadBatch` instances rather than concatenating them silently。

### `ResolveOptions`

```java
public record ResolveOptions(
        int maxRanges,
        boolean allowCache,
        boolean includeEntryIndex) {
}
```

`maxRanges` limits resolved object ranges, not logical records. `read` applies `ReadOptions.maxRecords`
when turning resolved ranges into payload bytes.

Phase 1 `resolve` always returns the `EntryIndexRef` stored in the selected offset index record because
that reference is part of the committed location. `includeEntryIndex` is reserved for future decoded entry
index materialization; Phase 1 must not read object footers or index objects inside `resolve` just because
this flag is true. Payload reads remain the responsibility of `read`/`WalObjectReader`.

### `ResolveResult`

```java
public record ResolveResult(
        StreamId streamId,
        long requestedOffset,
        List<ResolvedObjectRange> ranges,
        long resolvedEndOffset,
        long metadataVersion) {
}
```

`metadataVersion` is a cache validation hint: use the maximum Oxia key version observed among selected
offset index records and the stream-head record consulted by the resolver. It is not a stream
`commitVersion` and must not be used as an offset ordering coordinate.

`resolvedEndOffset` is the first logical offset not covered by `ranges` after applying `maxRanges` and any
read-side record limit. For an empty result it equals `requestedOffset`.

### `ResolvedObjectRange`

```java
public record ResolvedObjectRange(
        OffsetRange offsetRange,
        long generation,
        ObjectId objectId,
        ObjectKey objectKey,
        ObjectType objectType,
        long objectOffset,
        long objectLength,
        Checksum sliceChecksum,
        PayloadFormat payloadFormat,
        List<SchemaRef> schemaRefs,
        EntryIndexRef entryIndexRef,
        Optional<ProjectionRef> projectionRef,
        long commitVersion) {
}
```

`resolve` is a first-class API because Future 2, Future 4, Future 5 and caches can reuse it without
forcing a payload read.

`ResolvedObjectRange.objectOffset/objectLength` describe the full committed stream slice payload range
referenced by the offset index. They do not include object header/footer bytes or an external index object.
The range must be non-negative and `objectOffset + objectLength` must not overflow.
`sliceChecksum` is required; `WalObjectReader` uses it to verify the full resolved slice before clipping.
`schemaRefs` are copied from the selected offset index record.

## 7. Trim Domain

```java
public record TrimOptions(
        Duration timeout,
        String reason) {
}
```

`trim(streamId, beforeOffset)` updates the trim low-watermark stored in stream head:

```text
/streams/{streamId}/head.trimOffset
```

Phase 1 `trim` never deletes object bytes. It only changes metadata used by read and future GC.

Read behavior:

- `startOffset < trimOffset` completes exceptionally with `NereusException` code `OFFSET_TRIMMED`；
- `startOffset >= trimOffset` uses normal resolver logic。

## 8. Object and Index References in API

These types can live in `nereus-api` if exposed by public results, or in `nereus-object-store` if results
are made implementation-specific. Phase 1 recommends keeping small immutable reference types in
`nereus-api`.

```java
public record ObjectId(String value) {
}

public record ObjectKey(String value) {
}

public record ObjectRange(
        ObjectKey objectKey,
        long offset,
        long length) {
}

public enum ObjectType {
    MULTI_STREAM_WAL_OBJECT,
    INDEX_OBJECT,
    STREAM_COMPACTED_OBJECT,
    CURSOR_SNAPSHOT_OBJECT
}

public record EntryIndexRef(
        EntryIndexLocation location,
        Optional<ObjectId> objectId,
        Optional<ObjectKey> objectKey,
        Optional<byte[]> inlineData,
        long offset,
        long length,
        Checksum checksum) {
}

public enum EntryIndexLocation {
    INLINE,
    OBJECT_FOOTER,
    INDEX_OBJECT
}

public record Checksum(
        ChecksumType type,
        String value) {
}

public enum ChecksumType {
    CRC32C,
    SHA256
}
```

Checksum validation:

- `CRC32C` values use lowercase 8-character hexadecimal text；
- `SHA256` values use lowercase 64-character hexadecimal text；
- checksum strings are metadata/API representation. Binary object files may encode CRC32C as fixed-width
  numeric fields while exposing the same canonical text in API records.

```java
public record SchemaRef(
        String namespace,
        String id,
        long version) {
}

public record ProjectionRef(
        ProjectionType type,
        String value) {
}

public enum ProjectionType {
    VIRTUAL_LEDGER,
    ENTRY_INDEX,
    PROTOCOL_HINT
}
```

Validation:

- `SchemaRef.namespace` and `SchemaRef.id` must be non-blank UTF-8 strings；
- `SchemaRef.version >= 0`；
- `ProjectionRef.value` must be non-blank when present；
- maps used in API values should be defensively copied. Stored/encoded metadata maps must be sorted by
  UTF-8 key bytes so golden-byte tests are stable. M1 implements this through
  `MetadataCanonicalizer.canonicalStringMap`. `schemaRefs` are sorted by their canonical tuple through
  `MetadataCanonicalizer.canonicalSchemaRefs`; lists with semantic order, such as append entries and
  manifest slices, preserve that order.

`CURSOR_SNAPSHOT_OBJECT` and `STREAM_COMPACTED_OBJECT` are included as type boundaries. Phase 1 does not
implement cursor snapshots or compacted object writing.

`EntryIndexRef` location rules:

- `OBJECT_FOOTER`: `objectId` and `objectKey` may be empty, meaning the same object as the resolved
  payload range. If one is present, both must be present. `inlineData` must be empty and `length > 0`；
- `INDEX_OBJECT`: `objectId` and `objectKey` must identify the index object. `inlineData` must be empty
  and `length > 0`；
- `INLINE`: `inlineData` must contain the encoded entry index, `objectId/objectKey` must be empty, and
  `offset/length` must be zero。

Phase 1 writer must emit only `OBJECT_FOOTER`. `INLINE` and `INDEX_OBJECT` are API compatibility values;
Phase 1 reader must reject them with `UNSUPPORTED_FORMAT`.

| Code | Retriable | Meaning |
| --- | --- | --- |
| `INVALID_ARGUMENT` | no | caller supplied invalid options, offsets, batch shape, or unsupported value combination |
| `CANCELLED` | yes | operation was cancelled before reaching a known final state |
| `STREAM_NOT_FOUND` | no | stream metadata missing |
| `STREAM_NOT_ACTIVE` | depends | stream state does not allow the requested operation |
| `APPEND_SESSION_EXPIRED` | yes | caller must reacquire |
| `FENCED_APPEND` | yes | stale epoch/token rejected |
| `OFFSET_CONFLICT` | yes | committed end offset changed before commit |
| `BACKPRESSURE_REJECTED` | yes | append or read rejected before irreversible object IO because local resource limits are full |
| `TIMEOUT` | yes | configured operation timeout elapsed; append final state may be unknown if stream-head CAS was already sent |
| `STORAGE_CLOSED` | no | `StreamStorage` or one of its required clients is closing or closed |
| `OBJECT_UPLOAD_FAILED` | yes | object store write failed |
| `OBJECT_READ_FAILED` | yes | object range read failed for a transient or unknown reason |
| `OBJECT_NOT_FOUND` | yes | offset index points to an object that is currently unavailable or missing |
| `OBJECT_CHECKSUM_MISMATCH` | no | durable bytes do not match manifest/index |
| `OFFSET_TRIMMED` | no | requested offset is below trim low-watermark |
| `OFFSET_NOT_AVAILABLE` | yes | an operation that requires an existing committed offset targeted data beyond committed end |
| `READ_LIMIT_TOO_SMALL` | no | first readable entry is larger than `ReadOptions.maxBytes` |
| `METADATA_UNAVAILABLE` | yes | Oxia unavailable or timeout |
| `METADATA_CONDITION_FAILED` | yes | CAS conflict not classified above |
| `METADATA_INVARIANT_VIOLATION` | no | committed metadata is internally inconsistent |
| `READ_RESOLUTION_FAILED` | depends | resolver could not map offset to object range |
| `UNSUPPORTED_FORMAT` | no | payload format, compression, or entry-index location is not implemented in Phase 1 |

Async rule:

```text
Public async methods complete exceptionally with NereusException.
Validation errors before future creation are allowed only for null arguments and programmer mistakes.
Invalid user data that can be represented in an API value should complete the returned future with
`INVALID_ARGUMENT`.

Caller-side `CompletableFuture.cancel` is advisory. If cancellation happens after an append reached an
irreversible boundary, the internal operation may still complete and the durable outcome must follow the
same boundary rules as timeout. Service-side cancellation before an irreversible boundary should use
`NereusException(CANCELLED)`.

Normal `read` and `resolve` calls at or beyond the current committed end do not fail with
`OFFSET_NOT_AVAILABLE`; they return an empty result (`ReadResult.endOfStream=true` for `read`). The error
code is reserved for future strict APIs or internal helpers that cannot represent EOF as a normal result.
```

Implementation helper:

```java
static <T> CompletableFuture<T> failedFuture(ErrorCode code, String message, Throwable cause);
```

Exception mapping:

| Source | Condition | ErrorCode |
| --- | --- | --- |
| Object store put | timeout before known result | `TIMEOUT` or `OBJECT_UPLOAD_FAILED` with retriable=true |
| Object store put | duplicate key with `ifAbsent=true` | `OBJECT_UPLOAD_FAILED` unless idempotent same checksum is proven at metadata layer |
| Object store read/head | object missing | `OBJECT_NOT_FOUND` |
| Object store read | transient IO/permission/network error | `OBJECT_READ_FAILED` |
| Object store checksum validation | returned bytes do not match expected checksum | `OBJECT_CHECKSUM_MISMATCH` |
| WAL writer sizing | final encoded object length exceeds configured `maxObjectBytes` | `INVALID_ARGUMENT` |
| Metadata client | transport unavailable, deadline, reconnecting | `METADATA_UNAVAILABLE` |
| Metadata client | CAS/version condition failed and not classified | `METADATA_CONDITION_FAILED` |
| Metadata codec | corrupt bytes, wrong record type, checksum mismatch | `METADATA_INVARIANT_VIOLATION` |
| Metadata state machine | stale append epoch/token | `FENCED_APPEND` |
| Metadata state machine | committed end changed | `OFFSET_CONFLICT` |
| Resolver | no covering index below committed end | `METADATA_INVARIANT_VIOLATION` |

## 10. Immutability and Validation

All API records should:

- validate constructor arguments；
- defensively copy mutable collections；
- avoid exposing mutable `byte[]` where practical。

For `byte[]` payloads, Phase 1 can choose one of two approaches:

1. copy on API boundary for safety；
2. document ownership transfer to reduce copies。

Initial implementation should prefer safety. Performance tuning can revisit after the state machine works.
